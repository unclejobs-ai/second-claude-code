## 1. 핵심 컨셉: "AI 온보딩 챗봇 제로베이스 구축기"

**한 줄 요약**: OpenClaw + Claude Code + .md 지식베이스로 신입사원 온보딩 챗봇을 만드는 전 과정을 유튜브 시리즈로 공개하면서, 자연스럽게 B2B 전문성을 증명한다.

**핵심 가설**: "기술 역량을 투명하게 보여주는 것이 가장 강력한 B2B 영업이다."

### 2.2 핵심 설계 원칙

| 원칙 | 설명 | 이유 |
|------|------|------|
| **.md 파일 = 1급 시민** | HR 팀이 직접 마크다운 수정 → 챗봇 자동 반영 | IT 의존 제거가 최대 차별점 |
| **UI 역할 분리** | Slack = 신입 접점, 대시보드 = HR 관리 접점 | 통합 UI는 양쪽 다 어중간 |
| **SQLite 로컬 RAG** | OpenClaw 내장 SQLite + FTS5 + sqlite-vec | 외부 벡터 DB 없이 시작 가능 |
| **하이브리드 검색** | BM25 키워드 + 벡터 시맨틱 병행 | 정확도 15-20% 향상 (리서치 검증) |
| **채널별 세션 격리** | Slack 채널 ID별 독립 세션 | 부서 간 정보 분리 |

### 2.3 기술 스택

| 레이어 | 선택 | 대안 |
|--------|------|------|
| AI 에이전트 | OpenClaw (셀프호스팅) | — |
| LLM | Claude API (한국어 우수) | GPT-4o, DeepSeek |

### 2.4 검증된 수치 (이전 리서치)

- HR 질문의 **82%** 자동 응대 가능
- 온보딩 시간 **30% 단축**
- 반복 문의가 HR 담당자 업무시간의 **30-40%** 차지
- 하이브리드 검색 시 정확도 **15-20%** 향상
- 최적 청크 사이즈: **600-1200 토큰** (온보딩 문서 기준)

## 3. YouTube 시리즈 구조 — 8부작

### 3.1 콘텐츠 퍼널 매핑

```
Awareness (문제 인식)     →  EP1, EP2
Consideration (해결책 탐색)  →  EP3, EP4, EP5, EP6
Conversion (행동 유도)     →  EP7, EP8
```

### 3.2 에피소드 브레이크다운

| EP | 제목 (안) | 퍼널 | 타겟 | 핵심 |
|----|----------|------|------|------|
| **1** | HR 팀이 하루에 받는 질문들 (실제 데이터) | Awareness | HR 매니저 | Pain 공감, 문제 정의 |
| **2** | ChatGPT로 사내 챗봇 만들려다 실패한 이유 3가지 | Awareness | IT 담당자 | 기존 접근의 한계, 왜 다른 방법이 필요한가 |
| **3** | OpenClaw + Claude로 챗봇 아키텍처 설계하기 | Consideration | IT 리더 | 기술 신뢰, Claude Code 개발 과정 공개 |
| **4** | 마크다운 파일로 지식베이스 구축 — HR이 직접 | Consideration | HR + IT | **최대 차별점**: 비기술자 친화성 시연 |
| **5** | Slack 봇 연동 — 설치 없이 바로 쓰기 | Consideration | IT 담당자 | 통합 용이성, 실제 Slack 화면 |
| **6** | 웹 대시보드 개발기 (온보딩 진행률 추적) | Consideration | HR 매니저 | 가시성 + 관리 도구 |
| **7** | 실제 신규 입사자가 챗봇을 쓰는 순간 (라이브) | Conversion | 경영진 | ROI 증명, 15분 이내, 자막 필수 |
| **8** | 3개월 후 운영 리뷰 — 수치로 보는 효과 | Conversion | 경영진+HR | 재현 가능성, 한국 대기업 레퍼런스 |

### 3.3 에피소드 간 연결 장치

- 각 에피소드 말미: **다음 편 예고 + 아직 해결 안 된 문제** 제시
- GitHub 저장소 링크: 코드 공개 (핵심 프롬프트는 비공개)
- 에피소드별 **마크다운 템플릿 다운로드** (Lead Magnet)

### 3.4 콘텐츠 톤 가이드

**DO:**
- 구축 과정의 시행착오를 솔직하게 보여줌
- "왜 이 기술을 선택했는가" 의사결정 과정 설명
- 실패 → 수정 → 개선 흐름 (신뢰 구축의 핵심)
- 한국 기업 사례 자연스럽게 언급 (POSCO, KB카드, 삼성SDS)

**DON'T:**
- "저희가 이런 걸 만들어드릴 수 있습니다" 직접 영업 ❌
- 완성품만 보여주기 (과정이 핵심) ❌
- 포트폴리오식 나열 ❌
- "만들어드립니다" 톤 ❌

**시청자 자연 전환 흐름:**
> "나도 해보고 싶다" → "혼자 하기 어렵다" → "이 팀에 문의하자"

## 4. Build-in-Public 포지셔닝 — 3가지 신뢰 레이어

### 레이어 1: 기술 신뢰 (Technical Credibility)
- Claude Code로 실제 개발하는 화면 공개
- 코드 리뷰, 아키텍처 결정 과정
- OpenClaw 247K GitHub stars 자연스럽게 언급 (홍보 아닌 컨텍스트)

### 레이어 2: 도메인 신뢰 (Domain Credibility)
- HR/온보딩 Pain Point를 정확히 이해한다는 것을 보여줌
- 실제 고객 피드백 기반 수정 과정 공개
- 실패 에피소드: "처음에 이렇게 했다가 바꿨습니다"

### 레이어 3: 프로세스 신뢰 (Process Credibility)
- 엉클잡스 x 케이트리 협업 방식 공개
- 에이전트 워크플로우 자동화 전문성 시연
- "우리한테 맡기면 이렇게 진행됩니다"를 영업 없이 보여줌

## 5. B2B 전환 경로

```
YouTube 시청자
    │
    ▼
[Awareness] EP1-2 시청 → 구독
    │
    ▼
[Consideration] EP3-6 시청 → GitHub Star + 템플릿 다운로드 (이메일 수집)
    │
    ▼
[Conversion] EP7-8 시청 → "온보딩 자동화 가능성 진단" 체크리스트 (PDF)
    │                      → 30분 무료 기술 상담 (낮은 허들)
    │                      → LinkedIn 크로스포스팅
    ▼
[영업] 케이트리 영업팀 → 제안서 작성
```

**핵심 Lead Magnet**: "마크다운 온보딩 지식베이스 스타터 킷" — EP1 공개와 동시 GitHub 무료 공개
- HR 정책 마크다운 파일 10개 예시
- OpenClaw 설정 파일
- Slack 봇 연동 가이드
